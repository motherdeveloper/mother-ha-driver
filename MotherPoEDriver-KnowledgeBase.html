Introduction

What the Mother PoE driver is, what the integration does, and how a single Home Assistant box serves Home Assistant, Loxone and KNX installations.

What it is

The Mother PoE driver is a PoE+ powered LED driver with two 24 V outputs, each a 0–100 % dimmer. The Mother spots shift their colour automatically with the level — warm when dimmed, ~4000 K daylight at full — so there is only a brightness control, never a separate colour setting.

The driver talks CoAP over the network (UDP, port 5683, CBOR payloads). That is efficient and robust, but it is not a language that home-automation controllers speak directly. This project bridges that gap.

Architecture — Home Assistant as the hub

Home Assistant runs the integration, which speaks CoAP to the driver directly. From there it acts as a universal local hub: it discovers drivers, exposes them as lights, and makes them available in languages other systems understand — plain HTTP for Loxone, group addresses for KNX. A single always-on box, such as a Raspberry Pi, is enough to serve every system.

Where to start

  • Home Assistant users → Home Assistant setup. Often zero-touch.
  • Loxone installers → Loxone setup. You also need a Home-Assistant box on the LAN.
  • KNX installers → KNX setup. You need a KNX/IP gateway and a Home-Assistant box.

How it works

The path a command travels, from a control to a dimmed spot.

The chain

  1. A controller (HA app, Loxone, KNX) asks for a brightness.
  2. Home Assistant receives it and the integration encodes a CoAP/CBOR request.
  3. The driver fades the channel to the target level; the spot shifts colour automatically with the level.
  4. HA polls the driver (~10 s) to keep state in sync — the firmware has no push.

Good to know

Brightness only. Each channel is a 0–100 % dimmer. The warm→daylight shift happens in the spot itself, so there is nothing colour-related to send.

Mid-fade readback. While a channel fades, a status read returns the live level, not yet the target — so a value may briefly read e.g. 24 % on its way to 60 %.

Discovery

On a normal HA install that sits on the device's LAN, drivers are found automatically via DHCP (hostname mls25-*) and mDNS (_mother-life._udp) — true zero-touch. Discovery only works when HA shares the device's network; behind NAT (e.g. Docker Desktop on WSL2) you add the driver by IP instead.

Designed to grow

Today both channels are lights. The integration already models each channel with a type, so future firmware that reports a channel as an air-quality or motion sensor can be added without touching the lighting code.

Home Assistant setup

Install the integration, add a driver, control two dimmable lights.

Install the integration

Via HACS (recommended)

  1. In HACS → ⋮ → Custom repositories, add the repo URL, category Integration.
  2. Search for Mother PoE driver, download it, and restart Home Assistant.

Manual

Copy custom_components/mother_mls25/ into your HA config/custom_components/ folder and restart.

Add a driver

  • Auto: drivers appear under Settings → Devices & Services → Discovered → click Add.
  • Manual: Settings → Devices & Services → Add integration → Mother PoE driver → enter the driver's IP.

Each driver becomes a device with two lights, Channel 0 and Channel 1. HA's transition maps to the firmware fade rate.

Where's the brightness slider?

The dashboard tile shows only a toggle. Click the light's name to open it — the brightness slider is in that detail dialog.

Loxone setup

Loxone cannot speak CoAP, so Home Assistant is the bridge. The Miniserver drives plain-text HTTP endpoints with Virtual Outputs (control) and Virtual Inputs (status).

⚠ Before you start

Home Assistant must run on the same LAN as the Miniserver and the drivers, with the integration installed and each driver added.

Find your HA box's real LAN IP (Windows: ipconfig → IPv4 Address, e.g. 192.168.1.21). The examples below use HA-IP — replace it everywhere. Never use localhost (that is the Miniserver itself) and not the driver's IP.

Give each driver a fixed IP (DHCP reservation) so the link stays valid.

The endpoints

PurposeURL (GET)Returns
Read levelhttp://HA-IP:8123/api/mls25/level?ch=060
Set level (0–100)http://HA-IP:8123/api/mls25/set?ch=0&level=<v.0>OK
On / Offhttp://HA-IP:8123/api/mls25/on?ch=0 · …/off?ch=0OK

ch is 0 or 1. Add &host=192.168.1.158 to target a specific driver when you have several.

Step by step in Loxone Config

1

Virtual Output (the connection)

  • Periphery tree → Virtual Outputs → Add Virtual Output.
  • Address: http://HA-IP:8123
2

Dimming command (analog)

  • Right-click the Virtual Output → Add Virtual Output Command.
  • Command for ON: /api/mls25/set?ch=0&level=<v.0>
  • Leave it analog (uncheck "use as digital output").
  • Input value 1 = 0, Input value 2 = 100 (1:1 pass-through). Unit = %.

Why <v.0> and not <v>?

<v> can send a decimal like 48.0. The .0 forces a whole number. (Integration v0.3.1+ also accepts decimals, but <v.0> keeps it clean.)

3

A control element (the dim knob)

The command is only the output. You need a control wired into it. The simplest reliable option is an analog Virtual Input used as a slider:

  • Virtual Inputs → Add Virtual Input (the plain one, not the HTTP input).
  • Properties: analog (digital off), Min 0, Max 100, unit %.
  • On a programming page, draw a wire from the Virtual Input's output → the Virtual Output Command's input. In live mode the command value must follow the slider.
  • Assign the control a Room and a Category — without both, it will not appear in the app.

Prefer a real light tile?

Use a Dimmer block and wire its analog dim-value output (the one that follows the slider in live mode) to the command's input. Give it a Lighting category for an on/off + brightness tile. Watch out: the dimmer's connectors are easy to mix up — verify in live mode which output actually carries the value.

4

Status feedback (optional)

  • Virtual Inputs → Add Virtual HTTP Input.
  • Address: http://HA-IP:8123/api/mls25/level?ch=0, poll e.g. 5 s.
  • Add a command with command recognition \v (the whole response is the number).

Test it

From any PC on the LAN (replace HA-IP):

curl "http://HA-IP:8123/api/mls25/level?ch=0"      → a number
curl "http://HA-IP:8123/api/mls25/set?ch=0&level=30"  → OK, the spot dims

Then Save → upload in Loxone Config, and drive the slider from the app. Repeat steps 2–4 with ch=1 for the second channel.

KNX setup

The Mother PoE driver is not a KNX device, but Home Assistant's built-in KNX integration links the lights to KNX group addresses — no extra code.

⚠ You need

A KNX/IP gateway (router or interface), a Home-Assistant box on the LAN with this integration and the driver added, and HA's KNX integration configured against your gateway.

Two directions

  • KNX → driver (control): Home Assistant listens to the push-button's group addresses (knx_event) and an automation drives the light.
  • Driver → KNX (status): knx.expose puts the light's state back on the bus.

Example — control & status for channel 0

knx:
  event:
    - address: "1/1/0"   # on/off command
    - address: "1/1/1"   # dim command (0-100%)
  expose:
    - type: binary
      entity_id: light.mls25_192_168_1_158_channel_0
      address: "1/1/2"
    - type: percent
      entity_id: sensor.poe_driver_ch0_brightness
      address: "1/1/3"

template:
  - sensor:
      - name: "PoE driver ch0 brightness"
        unit_of_measurement: "%"
        state: >
          {% set b = state_attr('light.mls25_192_168_1_158_channel_0','brightness') %}
          {{ ((b | int(0)) / 255 * 100) | round(0) }}
- alias: "KNX -> PoE driver ch0 dim"
  trigger:
    - platform: event
      event_type: knx_event
      event_data: { destination: "1/1/1" }
  action:
    - service: light.turn_on
      target: { entity_id: light.mls25_192_168_1_158_channel_0 }
      data: { brightness_pct: "{{ trigger.event.data.data | int }}" }

Adapt to your installation

Replace the entity IDs and group addresses with your own. The knx_event payload field can vary by HA version — verify it live under Developer Tools → Events → listen to knx_event and adjust the templates. See docs/knx.md in the repo for the full recipe.

API endpoints

For reference — the HTTP endpoints HA exposes, and the CoAP endpoints the driver exposes.

HTTP (Home Assistant → controllers)

Plain text, no auth, GET. Base: http://HA-IP:8123.

Method · PathBody / paramsReturns
GET /api/mls25/levelch, optional host60
GET /api/mls25/setch, level 0–100, optional rate, hostOK
GET /api/mls25/on · /offch, optional hostOK

CoAP (driver firmware)

UDP port 5683, IPv4, CBOR payloads (content-format 60). No auth. No push (poll GET /led/<ch>).

Method · PathBody (CBOR)Effect
GET /led/<0..1>→ {"lvl":0..100}read level
POST /led/<0..1>{"lvl":0..100,"rate"?:0..100}set level + fade
POST /led/<0..1>/on · /off—on (100%) / off
POST /grp/<0..63>{"lvl":..,"mcast_fwd"?:bool}set whole group

Discovery

  • DHCP hostname mls25-<mac> — unique per device.
  • mDNS service _mother-life._udp.local, instance name MLS25.

Troubleshooting

Common issues during installation, and how to resolve each.

SymptomCause & fix
Driver not auto-discoveredHA isn't on the device's LAN (e.g. behind NAT in Docker Desktop / WSL2). Add it by IP instead, or run HA OS on a Pi/NUC on the LAN.
Loxone does nothingMost often the Virtual Output Address is wrong — it must be http://HA-IP:8123 with your HA box's real LAN IP (not a doc example, not localhost, not the driver IP).
Command rejected (ERR bad value)Loxone sent a decimal (48.0). Use level=<v.0> in the command, or update to integration v0.3.1+ which accepts decimals.
Slider moves but command value doesn'tThe wire into the Virtual Output Command isn't actually linked. Re-draw it from the source's output dot to the command's input dot; confirm in live mode the command value follows.
Control not visible in Loxone appThe element has no Room and/or Category. Assign both.
No brightness slider in HAThe tile shows only a toggle — click the light's name to open the detail dialog with the slider.
Level reads a "wrong" numberA read during a fade returns the live level, not the target. Read again once the fade completes.
Can't ping HA but it worksWindows blocks ICMP ping by default while still serving port 8123. Test with curl http://HA-IP:8123/api/mls25/level?ch=0, not ping.
Reachable from the host but not the MiniserverWindows Firewall may block inbound 8123 from other devices, or the network profile is "Public". Allow inbound TCP 8123 / set the profile to Private.
